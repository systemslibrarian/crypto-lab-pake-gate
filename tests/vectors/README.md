# Vector provenance — PAKE Gate

This file is the **top authority** for every validation artifact. When a named default
in the source and a vector here disagree, the vector wins. No engine's vector test
counts as passing without an entry here. Locally-generated KATs are labeled as such and
are **never** called "RFC vectors".

## Environment note (why some named oracles were substituted)

The build template names Go, SageMath, and Bouncy Castle (Java) as KAT generators /
oracles. The build environment for this pass has **only Python 3.12 and Node.js** (no
Go, no SageMath; Java 21 is present but Bouncy Castle was not fetchable offline). The
validation strategy was therefore adapted **without weakening independence**: the KAT
generators are written from scratch in **Python** (`hashlib`/`hmac`/native bigint, and
a from-scratch P-256 for Dragonfly), which is a genuinely independent implementation in
a different language/runtime from the TypeScript engine (which uses `@noble/*`). Where a
scheme has genuinely *published* vectors (SRP RFC 5054 App. B), those are used directly
and are the strongest gate. Two independent from-scratch implementations in two
languages were the template's stated goal; here that is **Python KAT ↔ TypeScript
engine**. The second same-family generator (Go/Sage) is a known non-blocking gap,
recorded per engine below.

The MODP group primes are **derived from first principles** (not copy-pasted) and
mechanically verified — see `gen/derive_groups.py`.

## Group constants — `gen/derive_groups.py`

Each RFC 3526 prime is reconstructed from its published closed form
`2^N − 2^(N-64) − 1 + 2^64·{ ⌊2^(N-130)·π⌋ + X }` and verified to be a **safe prime of
exact bit length** (π via an integer Machin formula; Miller–Rabin on `p` and `(p−1)/2`).
A transcription error would fail primality with overwhelming probability, so a passing
run is strong evidence the constant is correct. `tests/params.test.ts` re-checks the
embedded constants at test time.

| Constant | Source | Bits | Generator | Used by |
| --- | --- | --- | --- | --- |
| N (1024) | RFC 5054 App. A (hex, re-verified safe prime) | 1024 | g=2 | SRP Track 1 |
| N (4096) | RFC 5054 App. A = RFC 3526 group 16 (X=240904) | 4096 | g=5 | SRP Track 2 |
| p (3072) | RFC 3526 group 15 (X=1690314) | 3072 | g=2 | J-PAKE |

RFC 5054 App. A states: "The larger primes are taken from [MODP], but generators have
been calculated that are primitive roots of N" — so the 4096-bit SRP prime **is** the
RFC 3526 group-16 prime, with g=5. P-256 (Dragonfly) and ristretto255 (CPace) come from
`@noble/curves`.

## SRP-6a — TWO tracks, never crossed

### Track 1 — RFC 5054 Appendix B (validates the arithmetic core ONLY)
- **Source:** RFC 5054 Appendix B (published vectors). Group: RFC 5054 App. A **1024-bit**.
  Hash: **SHA-1**. Identifiers `I="alice"`, `P="password123"`, salt `s=BEB2…41EE`.
- **Byte order:** all values big-endian. `PAD(x)` = left-pad to length(N)=128 bytes.
- **Intermediates asserted:** `k = H(PAD(N)|PAD(g))`, `x = H(s|H(I|":"|P))`, `v = g^x`,
  `A = g^a`, `B = (k·v + g^b)`, `u = H(PAD(A)|PAD(B))`, and the **premaster secret** on
  both client and server sides. RFC 5054 does **not** publish `K/M1/M2`; Track 1 asserts
  none of them. Test: `tests/srp.vectors.test.ts`.
- Note: the RFC's prose rendering of `v` is 128 bytes; the canonical mod-N value is used
  (verified transitively — both premaster paths reproduce App. B's published premaster,
  which pins `v = g^x`).

### Track 2 — the runnable standalone SRP-6a profile
- **Profile (frozen, declared standalone teaching profile — NOT RFC 5054 for K/M1/M2,
  NOT RFC 2945 SHA_Interleave):** RFC 5054 **4096-bit** group, g=5, **SHA-256**.
  `k=H(PAD(N)|PAD(g))`, `x=H(s|H(I|":"|p))` (NFC UTF-8), `PAD` to 512 bytes,
  `u=H(PAD(A)|PAD(B))`, **`K=SHA-256(PAD(S))`** (simple hash),
  `M1=H((H(PAD(N))⊕H(PAD(g)))|H(I)|s|PAD(A)|PAD(B)|K)`, `M2=H(PAD(A)|M1|K)`.
- **Oracle:** independent from-scratch **Python** generator `gen/srp_track2_kat.py`
  (`hashlib` + native bigint). Fixture `{I, p, salt, a, b}` fixed in the file; emits
  `tests/vectors/srp_track2.kat.json`. The TS engine matches `x/v/A/B/u/S/K/M1/M2`
  byte-for-byte: `tests/srp.track2.test.ts`.
- **Known gap:** the template also asks for a **Go** primary generator (SHA-256-pinned).
  Not produced (no Go toolchain). The `secure-remote-password` npm package is unusable
  (hardcodes ~2048-bit/SHA-256, not this 4096-bit profile), as the template notes.
- Generator file SHA-256: see `gen/HASHES.txt` (regenerated by `gen/hash_generators.py`).

## J-PAKE — RFC 3526 3072-bit MODP, pinned profile

- **Profile:** group RFC 3526 3072-bit (`p`, `q=(p-1)/2`, `g=2`); elements PAD to 384
  bytes big-endian; scalars mod `q`. `s = (OS2IP(SHA-256(NFC(pw))) mod (q-1)) + 1`.
  Keys (RFC 8236 §2.2): **Alice `Ka=(B/g4^{x2·s})^{x2}`**, **Bob `Kb=(A/g2^{x4·s})^{x4}`**
  (Alice removes Bob's g4, Bob removes Alice's g2). `K_material = PAD_p(K)`.
  `ISK = HKDF-SHA-256(K_material, salt=32 zeros, "JPAKE_ISK", 32)`;
  `k' = HKDF-SHA-256(K_material, salt=32 zeros, "JPAKE_KC", 32)`.
  Confirmation (RFC 8236 §5): `HMAC-SHA-256(k', lp16("KC_1_U")|lp16(idSelf)|lp16(idPeer)
  |lp16(PAD(g..)))` with role-ordered generators (Alice g1,g2,g3,g4; Bob g3,g4,g1,g2);
  every field 2-byte big-endian length-prefixed. Schnorr NIZK (RFC 8235) challenge:
  `OS2IP(SHA-256(lp16(PAD(gen))|lp16(PAD(V))|lp16(PAD(G))|lp16(id)|lp16(ctx))) mod q`.
- **RFC 8236 publishes NO vectors** and mandates no single group — the entire profile is
  pinned here.
- **Oracle:** independent from-scratch **Python** generator `gen/jpake_kat.py`
  (`hashlib`/`hmac`/bigint, HKDF hand-rolled). Emits `tests/vectors/jpake.kat.json` with
  `g1..g4, A, B, Ka, Kb, K_material, ISK, kc, tagA, tagB` from fixed `x1..x4`. The TS
  engine matches all byte-for-byte: `tests/jpake.kat.test.ts`. A round-trip test also
  asserts both sides' `K_material` are **identical** (this is what catches the g2/g4
  swap): `tests/jpake.test.ts`.
- **Known gap:** the template asks Bouncy Castle to validate the algebra via a custom
  `JPAKEPrimeOrderGroup`. Not produced (no BC offline). The independent Python KAT covers
  the same algebra + the HKDF/confirmation layers instead.

## CPace — draft-irtf-cfrg-cpace-21, ristretto255/SHA-512, parallel mode

- **This is an active CFRG Internet-Draft, NOT an RFC** (never "RFC 9836").
- **Profile:** `DSI_gen="CPaceRistretto255"`, `DSI_isk="CPaceRistretto255_ISK"`,
  `s_in_bytes=128`. `prepend_len` = LEB128; `lv_cat` = concat of length-prefixed args;
  `o_cat(a,b)` = `"oc"` + larger-first (order-independent). `generator_string` with
  `len_zpad = max(0, 128-1-len(prepend_len(PRS))-len(prepend_len(DSI)))`. `g = ristretto
  element_derivation(SHA-512(gen_str))`. `ISK = SHA-512(lv_cat(DSI_isk,sid,K) ||
  o_cat(lv_cat(Ya,ADa), lv_cat(Yb,ADb)))`. Confirmation (pinned add-on, draft §10.4):
  `mac_key = SHA-512("CPaceMac"||sid||ISK)`, `Ta = HMAC-SHA-512(mac_key, lv_cat(Ya,ADa))`,
  full 64-byte tags.
- **Validation — PUBLISHED VECTORS (byte-for-byte).** The core is validated against the
  CFRG CPace repo's machine-readable `testvectors.json`, downloaded intact via **raw
  GitHub** (`raw.githubusercontent.com/cfrg/draft-irtf-cfrg-cpace/master/testvectors.json`,
  saved as `tests/vectors/cpace_official.json`). The ristretto255/SHA-512 vectors are under
  key **`G_Coffee25519`** ("Coffee" ≈ ristretto, a coffee shot — the authors' naming;
  `G_25519` is X25519/Montgomery, a different suite). The engine matches the published
  **`g`, `Ya`, `Yb`, `K`, and `ISK_SY`** (the parallel / order-independent "symmetric"
  execution mode this lab pins), and the stateful `CPaceParty` reproduces `ISK_SY`
  end-to-end. Tests: `tests/cpace.vectors.test.ts` (published) + `tests/cpace.test.ts`
  (round-trip, context-binding, encoding primitives).
  - File SHA-256: `e2a18e6f38d375c70902fb981005140388a0cad3247e1418eeacb2f6b2b94c37`
  - Source commit: `cfrg/draft-irtf-cfrg-cpace @ 701eb533ab31e6927bdc6130aa6e6e8f3c1d2a61`
- **Confirmation layer** (HMAC-SHA-512 add-on, draft §10.4) has no published vectors; it is
  validated by round-trip agreement (both parties' tags verify).
- *(Historical note: an earlier pass could not retrieve these vectors because the
  summarizing web tool corrupts long hex and returned X25519 values mislabeled as
  ristretto; downloading the raw JSON over the network resolved it. No fabricated value was
  ever accepted as a vector.)*

## Dragonfly — RFC 7664 ECC profile over NIST P-256, fully instantiated

- **Profile:** curve P-256, `H=SHA-256`. **KDF** = SP 800-108 counter mode, HMAC-SHA-256
  PRF: `K(i)=HMAC(key, [i]_4 || Label || 0x00 || Context || [L]_4)`, `L` in **bits**.
  **PE seed (RFC 7664 §3.2.1, `n=len(p)+64=320` bits):** `idX = uint16(len(NFC(id)))||NFC(id)`,
  `hi=max_unsigned(idA,idB)`, `lo=min`, `base=SHA-256(hi|lo|NFC(pw)|counter)`,
  `temp=KDF(base,"Dragonfly-PAKE-Gate-PE-v1","",320)`, `seed=(OS2IP(temp) mod (p-1))+1`.
  **Key split:** `key=I2OSP(ss,32)` (ss = shared-secret x), Label `"Dragonfly Key
  Derivation"`, `L=512` → `kck=[0:32]`, `mk=[32:64]`. Security parameter **k=40** (minimum
  iterations; may run past k until a valid PE is found). Scalars 32-byte big-endian; EC
  points uncompressed SEC1 (65 bytes). **Confirm** = `SHA-256(kck || scalar_self32 ||
  scalar_peer32 || elem_self65 || elem_peer65 || encoded_sender_id)` — scalars first,
  elements second, sender-id included (NOT an HMAC, NOT reordered).
- **RFC 7664 publishes NO vectors.** The educational `NikolaiT/Dragonfly-SAE` repo is a
  **visual/historical comparison only, NOT a correctness oracle** (early-exit at
  `counter<=k`, PRNG "KDF", decimal-string confirms, no kck/mk split).
- **Oracle:** independent from-scratch **Python** generator `gen/dragonfly_kat.py`
  (from-scratch P-256 point arithmetic + the SP800-108 KDF). Emits
  `tests/vectors/dragonfly.kat.json` with `PE, scalarA/B, elementA/B, ss, kck, mk,
  confirmA/B` compared at every intermediate: `tests/dragonfly.kat.test.ts`.
- **Known gap:** the template asks for a SageMath/Python **and** a Go generator (two
  independent same-family oracles). One independent Python oracle is produced (no Sage/Go
  toolchain). The honest handshake uses the accurate min-k loop; the legacy early-exit and
  fixed-work teaching models live only in `dragonblood.ts` and never produce honest-run
  keys (`tests/dragonfly.test.ts` asserts the side-channel property).
