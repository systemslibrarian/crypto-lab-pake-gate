# The PQ-PAKE seam: KEM ≠ group

A password-authenticated key exchange like the four in this lab runs a Diffie–Hellman inside a group: both peers derive a password-dependent group element and combine private scalars over it. A key-encapsulation mechanism (KEM) — the shape post-quantum primitives such as ML-KEM take — has **no group structure to run a DH over**; there is no scalar multiplication, no password-derived generator, nothing to "juggle." So a real PQ-PAKE cannot do PAKE *inside* the KEM. Instead, constructions like CAKE/OCAKE and Noise-KEM-PAKE keep the classic EKE idea and move it up a layer: they **encrypt or blind the KEM public key (and/or ciphertext) under a password-derived symmetric key**, then run the KEM normally, so an attacker who cannot guess the password cannot recover a usable public key to encapsulate against. Because the password now keys a *symmetric wrapper around* the KEM rather than a *generator within* a group, the extension seam in `src/pake/types.ts` consumes a `KemBackend` plus a `PasswordKeyedEncoding` (the EKE-style wrapper) — never a `DhGroup`. That type distinction is the seam: a PQ-PAKE engine is wired to `KemBackend + PasswordKeyedEncoding`, and the four current DH-group engines are wired to `DhGroup`, so the two families cannot be accidentally interchanged.
